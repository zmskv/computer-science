# Распределённый `grep` с кворумом

`mygrep` — это распределённая версия `grep`.
Один и тот же бинарный файл может работать в двух режимах:

- `worker`-узел: принимает задачи по HTTP и обрабатывает шарды параллельно
- координатор / CLI-утилита: читает файл или `stdin`, делит вход на шарды, отправляет их узлам, ждёт кворум одинаковых ответов по каждому шарду и печатает итоговый результат

## Как это работает

1. Координатор читает входные данные: файл, несколько файлов или `stdin`.
2. Вход делится на шарды — небольшие группы строк.
3. Каждый шард отправляется на все указанные узлы.
4. Узлы независимо выполняют поиск по своему полученному шарду.
5. Координатор принимает результат шарда только тогда, когда получит кворум одинаковых ответов.
6. После этого результаты всех шардов склеиваются в исходном порядке и выводятся в консоль.

Пример:

- есть 3 узла
- кворум = 2

Это значит, что для каждого шарда достаточно получить 2 одинаковых ответа из 3, чтобы считать результат подтверждённым.

## Сборка

```powershell
go build -o mygrep.exe ./cmd/mygrep
```

## Запуск worker-узлов

Откройте три терминала и запустите:

```powershell
.\mygrep.exe -serve -listen :8081
.\mygrep.exe -serve -listen :8082
.\mygrep.exe -serve -listen :8083
```

После этого у вас будут работать три узла, готовые принимать задания от координатора.

## Распределённый поиск

Поиск по файлу с кворумом `2` из `3` узлов:

```powershell
.\mygrep.exe `
  -pattern error `
  -F `
  -n `
  -peers http://127.0.0.1:8081,http://127.0.0.1:8082,http://127.0.0.1:8083 `
  -quorum 2 `
  app.log
```

Что здесь происходит:

- `-pattern error` — искать строки с `error`
- `-F` — искать как обычную подстроку, а не как регулярное выражение
- `-n` — выводить номер строки
- `-peers` — список worker-узлов
- `-quorum 2` — считать результат верным, если хотя бы 2 узла совпали

## Чтение из `stdin`

Можно искать не только по файлу, но и по входному потоку:

```powershell
Get-Content app.log | .\mygrep.exe `
  -pattern timeout `
  -peers http://127.0.0.1:8081,http://127.0.0.1:8082,http://127.0.0.1:8083
```

В этом случае координатор читает строки из стандартного ввода, разбивает их на шарды и отправляет узлам.

## Локальный режим

Если не указывать `-peers`, утилита работает как обычный локальный `grep`:

```powershell
.\mygrep.exe -pattern error -F -n app.log
```

В этом режиме сеть не используется: поиск выполняется внутри одного процесса.

## Поддерживаемые флаги

- `-pattern` — шаблон поиска
- `-F` — искать как фиксированную строку
- `-i` — игнорировать регистр
- `-n` — выводить номера строк
- `-v` — инвертировать поиск, то есть выводить несовпавшие строки
- `-peers` — список адресов worker-узлов через запятую
- `-quorum` — сколько одинаковых ответов нужно для подтверждения результата по шарду
- `-shard-size` — сколько строк включать в один шард
- `-timeout` — таймаут сетевого запроса к одному узлу
- `-serve` — запустить процесс в режиме worker-узла
- `-listen` — адрес, на котором worker слушает HTTP
- `-workers` — число goroutine внутри worker-узла для параллельной обработки задач

## Что проверяется тестами

Тесты проверяют:

- корректность обычного поиска
- работу распределённого режима
- достижение кворума даже при падении одного из узлов
- совпадение результата `mygrep` с результатом обычного `grep.exe`

Запуск тестов:

```powershell
go test ./...
```

## Сравнение с обычным `grep`

Обычный `grep`:

- читает входные данные
- ищет совпадения
- печатает результат

`mygrep` делает то же самое, но дополнительно:

- делит вход на шарды
- отправляет шарды нескольким узлам
- обрабатывает их параллельно
- использует кворум для подтверждения результата

То есть это не просто поиск по файлу, а распределённая и отказоустойчивая версия `grep`.
